# /update-qa

コードの変更内容を読み取り、`docs/requirements.md` と `docs/external-spec.md` を更新してコミットします。

## 実行手順

### 1. 変更内容の把握

以下を順番に確認する：

```bash
git log --oneline -5          # 最近のコミット履歴
git diff HEAD~1 HEAD          # 直前のコミットとの差分
```

### 2. バージョン番号の決定

`docs/requirements.md` のヘッダーから現在のバージョンを読み取り、以下のルールで次のバージョンを決める：

| 変更の種類 | 操作 | 例 |
|---|---|---|
| Quest（新ページ・大機能追加） | X を +1、Y・Z を 0 にリセット | 4.0.0 → 5.0.0 |
| 通常アップデート（機能改善・追加） | Y を +1、Z を 0 にリセット | 4.2.0 → 4.3.0 |
| バグ修正のみ | Z を +1 | 4.2.0 → 4.2.1 |

コミットメッセージに `Quest` が含まれる場合は Quest 扱い。

### 3. requirements.md の更新

- ヘッダーのバージョンと最終更新日（今日の日付）を更新する
- 変更した機能に対応する要件の記述を追加・修正する（3.x 節）
- 「7. 変更履歴」に新しい行を追記する

### 4. external-spec.md の更新

- ヘッダーのバージョンと対象バージョンを requirements.md に合わせる
- 新機能・変更機能に対応するテストケース（TC-XX）を追加する
  - 既存の最大 TC 番号の次の番号を使う
  - **目的**・**操作手順**・**期待結果** の3点セットで記述する
  - 期待結果は具体的に（「正しく動作する」ではなく「○○が表示される」）
- 「8. 変更履歴」に新しい行を追記する

### 5. コミット

```bash
git add docs/requirements.md docs/external-spec.md
git commit -m "docs: update QA docs to vX.Y.Z"
git push -u origin master
```

## 注意事項

- コードの変更がすでにコミット済みであることを前提とする
- docs の変更だけを別コミットにする（コードと混ぜない）
- TC 番号は external-spec.md 全体を検索して最大値を確認してから採番する
